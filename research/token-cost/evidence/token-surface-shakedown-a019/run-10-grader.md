Oversized discovery output uses CCR (Compress–Cache–Retrieve): the fully assembled, pre-token-truncation payload is retained in a per-session in-memory store, while the caller receives a line-truncated summary and a 12-hex-character retrieval handle.

## 1. Eligible tools and default budgets

The authoritative profile table is `TOOL_OUTPUT_PROFILES` in [src/protocol/ccr.rs:23](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/ccr.rs:23).

| Tool | Default `max_tokens` | Effective CCR scope |
|---|---:|---|
| `search_text` | 8,000 | All locally rendered search paths |
| `search_symbols` | 8,000 | Normal rendered results |
| `find_references` | 8,000 | `mode="references"` only |
| `explore` | 12,000 | Normal rendered results |
| `get_repo_map` | 16,000 | `detail="full"` only |

A caller-provided `max_tokens` takes precedence; otherwise `resolve_tool_max_tokens` uses the profile default ([src/protocol/ccr.rs:57](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/ccr.rs:57)). `apply_ccr_budget` checks eligibility and dispatches to the shared CCR limiter ([src/protocol/mod.rs:692](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/mod.rs:692)).

Two important mode exceptions:

- `get_repo_map` sends only `detail="full"` through CCR; compact/tree modes use ordinary token truncation and no profile default ([src/protocol/tools.rs:4297](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/tools.rs:4297)).
- `find_references(mode="implementations")` returns before the CCR call. Ordinary reference mode reaches CCR at the end of the handler ([src/protocol/tools.rs:8117](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/tools.rs:8117), [src/protocol/tools.rs:8363](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/tools.rs:8363)).

## 2. Complete result versus stored result

`enforce_token_budget_with_ccr` uses an approximate ceiling of `max_tokens × 4` bytes—there is no tokenizer involved ([src/protocol/ccr.rs:228](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/ccr.rs:228)).

- No positive budget: return the complete assembled payload unchanged.
- Payload at or below `budget × 4`: return it unchanged, with no continuation footer.
- Payload above the threshold: first preserve the complete payload, then produce a line-boundary-truncated summary. If that summary is genuinely shorter, store the complete payload and append the continuation footer.
- Defensive edge case: if truncation saved nothing (`summary.len() >= full.len()`), return the summary without storing or exposing a handle.

This ordering guarantees that retrieval returns the pre-token-truncation string, rather than a previously truncated copy ([src/protocol/ccr.rs:219](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/ccr.rs:219), `apply_ccr_overflow` at [src/protocol/ccr.rs:194](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/ccr.rs:194)).

“Complete” here means complete after upstream tool limits and shaping. For example, `search_text` first ranks and compacts to at most 20 files and 10 matches per file, then formats and applies CCR ([src/protocol/ccr.rs:12](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/ccr.rs:12), [src/protocol/ccr.rs:269](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/ccr.rs:269), [src/protocol/tools.rs:5316](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/tools.rs:5316)). Retrieval does not undo `limit`, per-file caps, or that earlier compaction.

## 3. Continuation exposure

The originating response ends with:

```text
---
CCR: full ranked output stored · retrieve: symforge_retrieve with hash="<12-hex-handle>"
```

That exact footer is generated by `apply_ccr_overflow` ([src/protocol/ccr.rs:212](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/ccr.rs:212)). It is appended after the budgeted summary, so the footer itself may make the final wire response slightly larger than the nominal ceiling.

Continuation is a dedicated read-only MCP tool, `symforge_retrieve`, taking one `hash` string ([src/protocol/tools.rs:10788](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/tools.rs:10788), `SymforgeRetrieveInput` at [src/protocol/read_tools.rs:396](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/read_tools.rs:396)).

## 4. Identifier validation and retrieval

`mint_handle` hashes the tool name plus the complete formatted payload with Rust’s `DefaultHasher`, masks it to 48 bits, and renders exactly 12 lowercase hexadecimal characters ([src/protocol/ccr.rs:186](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/ccr.rs:186)).

On retrieval, the handler:

1. Trims whitespace.
2. Lowercases the identifier.
3. Requires exactly 12 ASCII hexadecimal characters.
4. Looks it up in the current session’s `CcrStore`.
5. Returns the stored string verbatim, or an explicit invalid/unknown-or-expired error.

See `symforge_retrieve` at [src/protocol/tools.rs:10792](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/tools.rs:10792).

Validation is syntactic only: retrieval does not recompute the hash or validate the originating tool. The store is per-session and memory-only, bounded to 256 entries and 32 MiB, with oldest-entry eviction ([src/protocol/ccr.rs:100](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/ccr.rs:100), [src/protocol/ccr.rs:171](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/ccr.rs:171)). Thus “expired” means absent because of eviction or session loss/restart; there is no time-based TTL.

## 5. Usage accounting on retrieval

A successful `CcrStore::retrieve`:

- increments `retrieves` by one;
- adds the exact stored payload byte length to `bytes_retrieved`;
- uses saturating arithmetic;
- leaves the blob stored, so repeated successful retrievals count repeatedly.

Invalid or unknown identifiers do not change counters because lookup succeeds before either increment ([src/protocol/ccr.rs:158](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/ccr.rs:158)).

Retrieval does not change `offloads` or `bytes_stored`, record fetched files/symbols, bump frecency, or create durable STEL ledger accounting. CCR economics remain per-session. `context_inventory` exposes `ccr_bytes_retrieved` but not the internal retrieval count ([src/protocol/session.rs:564](/<temp>/symforge-token-shakedown-a10ff102/src/protocol/session.rs:564)); the admin compression heuristic likewise exposes bytes retrieved, offloads, and stored bytes ([src/server/admin/api_v1.rs:42](/<temp>/symforge-token-shakedown-a10ff102/src/server/admin/api_v1.rs:42)).
